When to use: an issue has been resolved and is likely to recur, or several customers have hit the same thing. Use this to capture the solution once so customers self-serve and agents stop re-solving it.

METHOD
1. Define the article's scope from the customer's words, not internal jargon. Title it as the problem the reader is searching for ("Export fails with 'file too large'"), so it surfaces in search.
2. Write a one-line summary and a "who this is for" note (plan, platform, version) so readers self-qualify fast.
3. List symptoms exactly as customers experience them — error text, screenshots, behavior — so they confirm they are in the right place.
4. Give the resolution as numbered, testable steps. One action per step, expected result after key steps, and note any prerequisites or permissions.
5. Add a "if that didn't work" branch and, where relevant, the root cause in plain language so readers understand rather than just paste.
6. Close with related articles and a contact path. Strip all internal-only detail (ticket IDs, engineer names, unshipped fixes).

OUTPUT FORMAT
- Title (search-friendly)
- Summary + applies-to (plan/platform/version)
- Symptoms
- Resolution (numbered steps)
- If that didn't work / root cause
- Related articles + contact
- Internal notes (kept separate, not published)
